# Context Map — межа static/dynamic контексту

Межа між «завантажується завжди» і «завантажується на вимогу» — архітектурне рішення, версійоване як код (whitepaper The New SDLC). Static = надійно, але дорого на кожен виклик; dynamic = дешево, але може бути «забуте». Помилка в один бік — спалені токени і розмитий сигнал, в інший — агент забуває правила безпеки.

## Static (завантажується завжди — тримати МІНІМАЛЬНИМ)

| Що | Де | Бюджет |
|----|----|--------|
| Frontmatter-описи skills (тригери) | SKILL.md description | 1–3 речення, лише тригери |
| Frontmatter-описи agents | agents/*.md description | 1–2 речення + example |
| Hard rules фабрики | README §Hard rules | ≤8 рядків |

## Dynamic (на вимогу — progressive disclosure)

| Що | Коли вантажиться |
|----|------------------|
| Тіло SKILL.md | збіг задачі з тригером |
| Паспорт петлі loops/*.md | strw-loop-run Step 1 |
| Конституція references/*.md | лише файли, названі у поточному skill |
| Рубрики/golden evals/ | checker phase і регресія |
| Шаблони templates/ | product-init |
| strw-state (стан, логи) | Step 2 Read state — лише релевантні файли продукту |

## Правила зміни межі

1. Додавання будь-чого у static — це PR-рішення: чому воно потрібне КОЖНОМУ виклику? Не потрібне → у dynamic.
2. Промпт агента ≤25 рядків; skill body ≤60 рядків; довше → винести в references/ і лінкувати.
3. Субагенти повертають компактні структури, не сирі дампи (subagent-delegation) — тримає контекст головного агента чистим.
4. L6 Retro раз на місяць перевіряє: чи не розрослися описи skills/agents (static-хвіст), чи не дублюються references.
5. Зміна цього файлу = зміна процесу → process-changelog.md.
