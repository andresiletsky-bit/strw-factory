# Eval Rubrics — планка це eval, а не демо

Checker оцінює артефакт за рубрикою, не «на око». Використання: (а) checker phase кожної петлі; (б) регресія після зміни промпту/скіла/паспорта (self-improvement.md); (в) калібрування нового checker'а проти golden-набору (`evals/golden/`).

## Два рівні перевірки (порядок фіксований)

0. **Структурний (детермінований):** `scripts/validate-artifact.sh <type> <file>` — обов'язкові секції контракту. FAIL → артефакт повертається maker'у БЕЗ виклику LLM-checker'а. Структурні фейли ніколи не йдуть в inbox.
1. **Змістовий (LLM-rubric):** нижче. Вердикт PASS лише коли всі виміри «прохідні».

## Спільні виміри (усі критичні артефакти)

| Вимір | Прохідний опис |
|-------|----------------|
| Task success | Артефакт розв'язує задачу стадії (не «схоже на звіт», а відповідає на питання gate) |
| Data integrity | Повний чекліст data-integrity-protocol.md: джерело/період/довіра для кожної цифри |
| **Trajectory compliance** | Trace maker'а відповідає паспорту: читав state, виконав обов'язкові перевірки, не виходив за scope, не пропускав кроки. Гарний вихід із пропущеними перевірками = FAIL |
| Аргументація | Кожне ключове твердження: твердження → доказ → джерело; контраргументи не замовчані |

## Per-artifact доповнення

- **idea-card:** сигнали попиту незалежні (≥2), ICE обґрунтований, не дубль Kill log.
- **validation-report:** TAM bottom-up перевірений, «конкурентів нема» розкрито, найдешевша перевірка реально найдешевша, рекомендація випливає з даних (не з тону).
- **prd:** метрики успіху вимірювані (число+період), НЕ-цілі змістовні, AC тестопридатні (стануть тестами ДО коду), tracking events покривають метрики успіху.
- **build-report:** diff vs PRD чесний (недороблене назване), тести написані до/разом з кодом і покривають AC+edge cases, security-секція заповнена (dep-audit, верифікація залежностей), CI-лінк живий.
- **growth-report:** цифри лише з metrics.md/PRD, чернетки без вигаданих тверджень.
- **portfolio-brief:** метрики фабрики повні (включно з first-pass accept rate і вартістю петель з loops-log), фокуси actionable, тренди чесні (2 точки ≠ тренд).

## Регресія (після будь-якої зміни промпту/скіла/паспорта)

Прогнати змінений компонент на відповідних golden-артефактах: очікуваний вердикт checker'а для golden = PASS; для golden із навмисними дефектами (розділ «сіді-кейси» у файлі golden) = FAIL із виявленням саме цього дефекту. Регресія не пройшла → зміна не застосовується, версія не бампається.
